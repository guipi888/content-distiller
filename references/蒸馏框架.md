# 蒸馏框架参考

## 一、什么是「方法论蒸馏」

方法论蒸馏 = 从一段内容（课程/视频/文章/对话）中，提取出**可复用的操作框架**，剥离掉具体案例和冗余叙述，只保留「怎么做」的核心步骤，并封装成体系化的技能包。

### 蒸馏 vs 摘要 的区别

| 维度 | 摘要 | 蒸馏 |
|------|------|------|
| 目的 | 知道讲了什么 | 能直接拿来用 |
| 输出 | 内容缩写 | 体系化技能包（方法论+参考资料+实例+模板） |
| 保留 | 核心观点 | 可执行的动作 |
| 丢弃 | — | 案例细节、背景铺垫、个人故事 |
| 检验 | 能复述内容 | 能照着做，且有实例可参照 |

## 二、五层蒸馏模型

从一段内容中逐层提取，越深层越有价值：

### Layer 1：信息层（最浅）
- 提取事实、数据、引用
- 价值：低（信息会过时）
- 示例：「某课程说小红书最佳发布时间是晚上8点」

### Layer 2：技巧层
- 提取具体操作技巧、快捷方式
- 价值：中（可直接用，但场景有限）
- 示例：「用 anytocopy.com 可以无水印下载小红书图片」

### Layer 3：方法论层（核心目标）
- 提取完整操作框架、SOP、决策模型
- 价值：高（跨场景复用）
- 示例：「内容选题三问法：写给谁看？解决什么问题？为什么是你来说？」

### Layer 4：心法层
- 提取底层思维模型、判断原则
- 价值：极高（指导一切决策）
- 示例：「先完成再完美，先跑通再优化」

### Layer 5：元认知层（最深）
- 提取「如何思考」的方法
- 价值：终极（学会学习）
- 示例：「遇到新领域，先找3个从业者的踩坑总结，比看10篇科普文有效」

**蒸馏策略**：优先提取 Layer 3-5 的内容。Layer 1-2 作为方法论的佐证保留，不单独成技能。

## 三、方法论识别信号

内容中出现以下信号时，大概率包含可蒸馏的方法论：

| 信号 | 说明 | 示例 |
|------|------|------|
| 步骤序列 | 「第一步…第二步…第三步…」 | 任何教程类内容 |
| 对比框架 | 「做之前 vs 做之后」「A方法 vs B方法」 | 经验分享类 |
| 检查清单 | 「必须包含…」「不能出现…」 | 规范类内容 |
| 决策树 | 「如果…就…；如果…就…」 | 策略类内容 |
| 公式/模型 | X = A + B + C | 方法论总结 |
| 反直觉判断 | 「大多数人以为…其实…」 | 认知类内容 |
| 踩坑复盘 | 「我犯了…错误，后来发现…」 | 经验类内容 |
| 重复模式 | 同一做法在多个场景被提及 | 验证过的规律 |

## 四、体系化技能包结构（核心规范）

每个蒸馏出的方法论技能，必须是一个**完整的体系化包**，而非单文件。

### 4.1 标准目录结构

```
方法论-{名称}/
├── SKILL.md                          # 必需 — 方法论主文档（核心）
├── references/                        # 必需 — 参考资料目录
│   ├── 原始内容存档.md                #   原始内容的完整存档（去噪后，含来源链接）
│   ├── 理论原理.md                    #   方法论的底层原理详解（为什么有效）
│   ├── 相关方法论对照.md              #   与同类方法论的对比分析（至少对比2个）
│   └── 局限性分析.md                #   这个方法论的盲区、失效场景、反对观点
├── examples/                          # 必需 — 实例目录
│   ├── 实例-01-成功.md               #   完整成功案例（输入→过程→结果，含数据）
│   ├── 实例-02-成功.md               #   第二个成功案例，来自不同场景
│   ├── 实例-03-失败.md               #   失败案例：用错了导致失败
│   └── 实例-04-边界.md               #   边界案例：做了一半发现不适用，中途切换
├── templates/                         # 可选 — 模板目录
│   ├── 模板-01.md                     #   可直接复用的模板
│   └── 检查清单.md                    #   操作检查清单（checklist）
└── scripts/                           # 可选 — 脚本目录（如方法论涉及自动化）
    └── helper.py                      #   辅助脚本
```

### 4.2 各文件职责详解

| 文件 | 是否必需 | 内容说明 | 质量标准 |
|------|---------|---------|---------|
| `SKILL.md` | ✅ 必需 | 方法论的完整说明，含前置条件提醒 | 照着SKILL.md就能直接执行，不需要其他文件 |
| `references/原始内容存档.md` | ✅ 必需 | 原始内容去噪后的完整存档，含来源链接、作者信息 | 保留方法论相关的所有上下文 |
| `references/理论原理.md` | ✅ 必需 | 方法论为什么有效的深度解释，引用相关理论 | 不只是「有效」，要说清「为什么有效」 |
| `references/相关方法论对照.md` | ✅ 必需 | 同类方法论对比表：这个方法 vs 其他方法的差异、优劣 | 至少对比2个同类方法 |
| `references/局限性分析.md` | ✅ 必需 | 盲区分析：什么时候失效、反对观点、作者自己说的局限 | 必须包含评论区/弹幕里的质疑声音 |
| `examples/实例-01-成功.md` | ✅ 必需 | 完整成功案例：背景→输入→操作过程→结果→复盘 | 必须包含具体数据/数字/效果 |
| `examples/实例-02-成功.md` | 推荐有 | 第二个成功案例，来自不同场景 | 展示方法论的跨场景适用性 |
| `examples/实例-03-失败.md` | ✅ 必需 | 失败案例：错误场景→错误操作→失败结果→错误分析→教训 | 帮助理解适用边界 |
| `examples/实例-04-边界.md` | ✅ 必需 | 边界案例：做了一半发现这个方法不适用，中途切换方法 | 教用户「在过程中感知到边界」 |
| `templates/模板-01.md` | 视情况 | 如果方法论涉及固定格式输出，提供模板 | 可直接复制使用 |
| `templates/检查清单.md` | 视情况 | 操作检查清单（checklist） | 逐项可勾选 |

### 4.3 SKILL.md 前置条件规范

每个 `SKILL.md` 必须在正文开头加入前置条件区块，格式如下：

```markdown
> ⚠️ 前置条件
> - 使用这个方法论之前，你需要已经具备：{能力/资源/前置方法论}
> - 如果你还没有，建议先做：{前置方法论名称}
> - 不适用场景：{一句话说明}
```

**触发规则**：
- 如果方法论依赖某个前置能力/资源 → 必须写
- 如果方法论有「没有XX就做不了」的限制 → 必须写
- 如果方法论是 Layer 4 心法类 → 必须标注「建议先掌握对应的 Layer 3 方法论」

### 4.4 实例写作规范

每个实例文件名统一为 `实例-{序号}-{类型}.md`，类型固定为「成功」「失败」「边界」三种。

#### 成功实例规范

```markdown
# 实例：{实例标题}（成功）

## 前置状态
- 使用前的状态：{起点}
- 已具备的条件：{什么已经准备好了}

## 输入
- 可用资源：{当时有什么}
- 限制条件：{时间/人力/工具的限制}

## 操作过程（按方法论步骤逐一记录）
### Step 1：{步骤名}
- 实际操作：{具体做了什么}
- 花费时间：{多久}
- 遇到的问题：{如有}
- 解决方式：{如有}

（后续步骤…）

## 结果
- 直接结果：{产出了什么}
- 数据/效果：{可量化的结果}
- 对比变化：{使用前 vs 使用后}

## 复盘
- 做得好的：{什么操作最有效}
- 可改进的：{下次怎么做更好}
- 关键经验：{一句话总结}
```

#### 失败实例规范

```markdown
# 实例：{标题}（失败）

## 错误场景
{什么时候、什么情况下用错了这个方法论}

## 错误操作
{具体做了什么错误的事}

## 失败结果
{导致了什么后果}

## 错误分析
- 根本原因：{为什么会错}
- 违反了方法论的哪条原则：{具体指出}
- 正确做法应该是：{应该怎么做}

## 教训
{一句话总结，防止下次再犯}
```

#### 边界实例规范（新增）

```markdown
# 实例：{标题}（边界）

## 场景
{做到什么阶段时，发现这个方法论不适用了}

## 已执行的部分
{已经按方法论做了哪些步骤，做到哪一步时意识到的}

## 为什么不适用
- 触发信号：{什么现象让你意识到「这个方法现在不合适」}
- 根本原因：{为什么会在这里失效}
- 该方法论的哪条边界被触发了：{引用 SKILL.md 里的适用边界}

## 切换方案
- 切换到的方法论：{名称}
- 切换后的效果：{结果}

## 经验
{一句话总结：下次遇到类似场景，如何提前判断要不要用这个方法}
```

### 4.5 局限性分析写作规范

`references/局限性分析.md` 是区分「真方法论」和「鸡汤」的关键文件，必须包含以下四个维度：

```markdown
# 局限性分析：{方法论名称}

## 1. 失效场景
{在什么情况下这个方法论完全无效，或者有严重副作用}
（至少写2个场景）

## 2. 盲区
{这个方法论没有覆盖到的地方，或者隐含的假设是什么}
（每个方法论都有隐含假设，必须挖出来）

## 3. 反对观点
{原作者自己提到的局限}（如有）
{评论区/弹幕/其他来源中，对这个方法的质疑}
（必须主动去找，不能只有原作者的一面之词）

## 4. 前置依赖
{用这个方法论之前，必须先具备什么}
{如果前置依赖不满足，会发生什么}
```

## 五、蒸馏质量标准

一个合格的体系化方法论技能包必须满足：

### 5.1 可执行性
- [ ] SKILL.md 有明确的输入条件（什么时候用）
- [ ] SKILL.md 有明确的操作步骤（怎么做）
- [ ] SKILL.md 有明确的输出结果（用完得到什么）
- [ ] SKILL.md 开头有前置条件区块（`> ⚠️ 前置条件`）
- [ ] 不依赖特定工具/平台（或明确标注依赖）

### 5.2 完整性
- [ ] 包含「为什么有效」的原理解释（references/理论原理.md）
- [ ] 包含局限性分析（references/局限性分析.md）
- [ ] 包含至少1个正面成功实例（examples/实例-01-成功.md）
- [ ] 包含失败实例（examples/实例-03-失败.md）
- [ ] 包含边界实例（examples/实例-04-边界.md）
- [ ] 包含同类方法论对照（references/相关方法论对照.md）

### 5.3 独立性
- [ ] 不需要回看原始内容就能使用
- [ ] 方法论本身自包含，不引用外部未解释的概念
- [ ] 可以被其他技能调用作为子流程

### 5.4 体系化
- [ ] 目录结构完整（SKILL.md + references/ + examples/）
- [ ] references/ 至少4个文件（原始存档+理论原理+对照分析+局限性分析）
- [ ] examples/ 至少4个文件（2成功+1失败+1边界）
- [ ] 成功实例包含具体数据，不是泛泛而谈
- [ ] 失败实例和边界实例帮助理解边界
- [ ] 原始内容已存档可追溯

## 六、方法论分类体系

蒸馏出的方法论按以下分类归档：

| 分类 | 代码 | 说明 | 示例 |
|------|------|------|------|
| 内容创作 | CC | 选题/写作/剪辑/发布 | 选题三问法 |
| 流量增长 | TG | 起号/涨粉/引流/裂变 | 标签矩阵法 |
| 变现转化 | MC | 定价/销售/转化/复购 | 阶梯定价法 |
| 效率系统 | ES | 自动化/工具链/工作流 | 批处理工作法 |
| 认知思维 | CT | 决策/学习/反思/成长 | 逆向思维法 |
| 技术实操 | TP | 具体工具操作/配置 | 扣子Bot搭建 |
| 运营管理 | OM | 团队/项目/数据/复盘 | 周复盘框架 |

## 七、反模式：不应该被蒸馏的内容

| 类型 | 原因 | 处理方式 |
|------|------|---------|
| 纯个人信息 | 不可复用 | 跳过 |
| 平台特定时效信息 | 会过时 | 记入知识库但不创建技能 |
| 未经验证的猜测 | 可能有害 | 标注「未验证」 |
| 常识性内容 | 价值低 | 跳过 |
| 矛盾的方法论 | 需要验证 | 保留两方观点，标注「待验证」 |
